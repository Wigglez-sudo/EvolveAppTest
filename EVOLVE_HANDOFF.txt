CONTEXT — a monthly food-DB run built all 4 shops to target (10,000
foods) but FAILED at the validate step: two Asda items came in at 928
and 923 kcal/100 g. Because validate ran BEFORE commit and exits 1 on
any error, the whole ~55-min build was discarded and nothing reached
main. (Main still held an older partial set: 721 foods.) Root cause was
a limit MISMATCH: the build accepted kcal up to 1000, the validator
rejected above 900 — so the build kept writing items CI would always
reject. 900 is correct (pure fat = ~900 kcal/100 g is the physical max;
928/923 are bad OFF data — a per-pack/serving figure mislabelled as
per-100 g).

FIXES
A. BUILD now drops bad data at the source (tools/build-openfoodfacts-
   uk-db.js, mapProduct): kcal ceiling 1000 → 900 to MATCH the
   validator, plus a finite-number guard (rejects NaN/Infinity) so a
   written file can never contain anything the validator rejects. Keep
   the two scripts' kcal/macro limits in sync if either is changed.
B. VALIDATOR (tools/validate-food-db.js): now checks EVERY item, not
   just the first 2000 per shop (a 2500-item file previously left a
   500-row blind spot). On error it prints the exact file+index, e.g.
   "asda.json[293]", so a flagged row can be found and deleted by hand.
   Added an optional --report (or EVOLVE_VALIDATE_REPORT=1) mode that
   lists problems but exits 0, for a local dry-run.
C. WORKFLOW reordered to BUILD → COMMIT → VALIDATE
   (.github/workflows/update-food-database.yml). Commit now runs BEFORE
   validate, so a long build is saved no matter what. Validate runs last
   as a HARD check: if anything slips through, the run goes RED (so the
   owner notices) but the data is already committed — fix is to edit the
   committed food-db/<shop>.json, delete the flagged row(s), commit.
   This is the owner-chosen design: "drop bad items during build" PLUS a
   loud-but-non-destructive backstop.
D. NODE 24 / ACTION DEPRECATION: bumped actions/checkout v4 → v5 and
   actions/setup-node v4 → v5 (both run on Node 24 natively) and removed
   the now-unnecessary FORCE_JAVASCRIPT_ACTIONS_TO_NODE24 env stopgap.
   Clears the "Node.js 20 actions are deprecated" warning for this
   workflow (GitHub forces Node 24 from 16 Jun 2026).

GITHUB PAGES (separate, harmless): the owner also saw a Node-20
deprecation warning on the Pages build. That is GitHub's OWN managed
"pages-build-deployment" workflow (Pages source = "Deploy from a
branch"); it has no file in the repo and the flagged checkout@v4 /
upload-artifact@v4 are nested INSIDE GitHub's upload-pages-artifact@v3.
It is not the owner's to fix, the deploy still succeeds, and GitHub will
update it. OPTIONAL: a new deploy-pages.yml is provided (NOT yet added
to .github/workflows/) that deploys the static PWA via current Node-24
actions (checkout@v5, configure-pages@v5, upload-pages-artifact@v4,
deploy-pages@v4). To use it: add it under .github/workflows/ AND set
repo Settings → Pages → Source to "GitHub Actions". This also skips the
needless Jekyll step (Evolve is plain static files). Left optional
because nothing is broken without it.

VALIDATION (this maintenance pass): both scripts pass `node --check`;
the build filter was unit-tested to drop the exact 928/923 items and a
non-finite value while keeping normal foods; the validator was tested to
catch a bad row at index 2200 (beyond the old 2000 cap) and to exit 0 in

WHAT EVOLVE IS
--------------
A single-user, offline-first PWA gym + nutrition tracker. Plain
HTML/CSS/JS — no framework, no build step, no backend, no accounts.
All state lives in the phone's localStorage. Installs to the home
screen on iPhone/iPad and Android and runs fully offline.
Tagline: "Train smarter · Become next."

5 tabs (Home / Train / Fuel / Progress / More) plus a full-screen
live-workout overlay. Databases: ~92 machines, 53 free weights,
58 home moves, ~40 cardio, 726 foods.

KEY CONVENTIONS
---------------
- Storage keys: main data = "evolve_v1"; in-progress live session =
  "evolve_live_v1". load() merges saved data over DEFAULT_DATA then
  migrate(). save() and persistLive() are try/catch wrapped.
- Units stored canonically (kg, kcal); converted on display
  (eVal/eUnit/liftFromKg, energy can be kcal or kJ).
- Targets: computeTargets(p) uses Mifflin–St Jeor. GOALS =
  {lose:-500, maintain:0, gain:+350}. prefs.targetMode = "auto" |
  "manual".
- Modals: openModal(html, opts) injects a sticky top bar with a grab
  handle and a ✕ (#modalX) unless opts.mandatory. closeModal handles
  the phone Back button via history.pushState/popstate. A module-level
  `liveCardioStop` lets closeModal stop a running cardio timer.
- Health figures are estimates, not medical advice (already noted in
  app + README). Keep that framing for any nutrition/macro features.
